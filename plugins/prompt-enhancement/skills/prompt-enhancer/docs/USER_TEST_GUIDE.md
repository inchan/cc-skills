## 테스트 목적

prompt-enhancer 스킬의 6개 핵심 영역이 올바르게 작동하는지 검증합니다:

| 영역 | 검증 항목 |
|------|-----------|
| 적응형 컨텍스트 | 프로젝트 정보 자동 감지 |
| GOLDEN 프레임워크 | 6요소 구조화 |
| 다중 목표 최적화 | 복잡도별 템플릿 적용 |
| 멀티에이전트 협력 | 스킬 체인 제안 |
| 도메인 패턴 | 기술 스택별 패턴 참조 |
| 위험 평가 | 잠재적 문제 식별 |

---

## 사전 준비

### 환경 설정

1. Claude Code가 설치되어 있어야 합니다
2. 테스트할 프로젝트 디렉토리에서 실행합니다
3. prompt-enhancer 스킬이 활성화되어 있어야 합니다

### 스킬 호출 방법

```bash
# 방법 1: 직접 호출
prompt-enhancer 스킬을 사용해서 다음 프롬프트를 향상시켜줘:
[테스트 프롬프트]

# 방법 2: 자동 감지 (skill-rules.json 등록 시)
다음 프롬프트를 향상시켜줘:
[테스트 프롬프트]
```

## 테스트 시나리오

### 시나리오 1: Simple (복잡도 < 0.3)

#### 입력 프롬프트

```
로그인 버튼 색상을 파란색으로 변경해줘
```

#### 예상 결과

**포함되어야 할 요소:**
- [ ] GOLDEN 요소 중 Goal, Output만 포함
- [ ] Layer 1 (컨텍스트)만 제공
- [ ] 프로젝트의 UI 프레임워크 감지
- [ ] 기존 버튼 파일 위치 참조

**예상 출력 형태:**

```markdown
## 향상된 프롬프트

### 목표 (Goal)
로그인 버튼의 배경색을 파란색(#0066CC 또는 프로젝트 디자인 시스템 기준)으로 변경

### 출력 (Output)
- 수정된 버튼 컴포넌트 코드
- 기존 hover/active 상태 스타일 유지

### 프로젝트 컨텍스트
- UI 프레임워크: [자동 감지 - React/Vue/Angular 등]
- 버튼 파일 위치: [자동 감지]
- 디자인 토큰/테마 파일: [자동 감지]
```

**예상 소요 시간**: 10-30초

---

### 시나리오 2: Standard (복잡도 0.3-0.7)

#### 입력 프롬프트

```
사용자 프로필 페이지에 프로필 이미지 업로드 기능 추가해줘
```

#### 예상 결과

**포함되어야 할 요소:**
- [ ] GOLDEN 6요소 전체 (Goal/Output/Limits/Data/Evaluation/Next)
- [ ] Layer 1-3 전체 (컨텍스트/지시/검증)
- [ ] 위험 평가 (High/Medium/Low)
- [ ] 구조화된 구현 단계

**예상 출력 형태:**

```markdown
## 향상된 프롬프트

### GOLDEN 프레임워크

| 요소 | 내용 |
|------|------|
| **Goal** | 프로필 페이지에서 사용자가 이미지를 선택/업로드하고 미리보기 후 저장 가능 |
| **Output** | 이미지 업로드 컴포넌트, API 엔드포인트, 파일 저장 로직 |
| **Limits** | 파일 크기 5MB 이하, 허용 형식 JPG/PNG/WebP, 업로드 시간 < 3초 |
| **Data** | [기존 프로필 컴포넌트], [파일 업로드 유틸리티], [스토리지 설정] |
| **Evaluation** | 업로드 성공률 100%, 미리보기 즉시 표시, 에러 핸들링 완비 |
| **Next** | 실패 시 재시도 로직, 이미지 크롭 기능은 후속 작업 |

### Layer 1: 컨텍스트
- 프로젝트 구조: [자동 감지]
- 기존 패턴: [파일 업로드 패턴 참조]
- 의존성: [multer, sharp, S3 등]

### Layer 2: 구조화된 지시
1. 이미지 선택 UI 컴포넌트 생성
2. 미리보기 기능 구현
3. 업로드 API 엔드포인트 추가
4. 파일 검증 및 저장 로직
5. 에러 핸들링 및 사용자 피드백

### Layer 3: 검증 기준
- 테스트 커버리지 > 80%
- 린트 에러 0개
- 보안: 파일 타입 검증, 크기 제한

### 위험 평가
| 수준 | 항목 | 설명 |
|------|------|------|
| Medium | 파일 저장소 | 클라우드 스토리지 설정 필요 |
| Low | 스타일 일관성 | 기존 컴포넌트와 디자인 통일 |
```

**예상 소요 시간**: 30-60초

### 시나리오 3: Extended (복잡도 > 0.7)

#### 입력 프롬프트

```
실시간 알림 시스템 구현해줘. 웹소켓으로 서버에서 푸시하고 클라이언트에서 토스트로 표시
```

#### 예상 결과

**포함되어야 할 요소:**
- [ ] GOLDEN 6요소 전체 + 상세 제약조건
- [ ] Layer 1-3 전체 + Phase 분리
- [ ] 멀티에이전트 협력 패턴 제안
- [ ] 자동 개선 루프 결과 (평가 점수)
- [ ] 상세 위험 평가

**예상 출력 형태:**

```markdown
## 향상된 프롬프트

### GOLDEN 프레임워크

| 요소 | 내용 |
|------|------|
| **Goal** | 서버 이벤트를 실시간으로 클라이언트에 전달하고 토스트 UI로 표시 |
| **Output** | WebSocket 서버, 클라이언트 연결 훅, 토스트 알림 컴포넌트, 알림 타입 정의 |
| **Limits** | 동시 연결 1000명, 메시지 지연 < 100ms, 재연결 자동화 |
| **Data** | [기존 인증 시스템], [토스트 라이브러리], [이벤트 소스] |
| **Evaluation** | 연결 안정성 99.9%, 메시지 전달률 100%, 메모리 누수 없음 |
| **Next** | 알림 히스토리 저장, 알림 설정 UI는 2차 작업 |

### Layer 1: 컨텍스트
- 아키텍처: [모노레포/마이크로서비스]
- 기존 실시간 패턴: [Socket.io/ws 사용 여부]
- 상태관리: [Redux/Zustand/Context]

### Layer 2: 구조화된 지시

#### Phase 1: 백엔드
1. WebSocket 서버 설정 (ws/Socket.io)
2. 인증 미들웨어 연동
3. 이벤트 타입 및 메시지 스키마 정의
4. 브로드캐스트/유니캐스트 로직

#### Phase 2: 프론트엔드
1. WebSocket 연결 관리 훅
2. 자동 재연결 로직
3. 토스트 알림 컴포넌트
4. 알림 타입별 스타일링

#### Phase 3: 통합
1. E2E 연결 테스트
2. 부하 테스트
3. 에러 시나리오 처리

### Layer 3: 검증 기준
- 단위 테스트 + 통합 테스트
- 연결 끊김 시 재연결 확인
- 메모리 누수 테스트

### 위험 평가

| 수준 | 항목 | 설명 | 완화 방안 |
|------|------|------|-----------|
| High | 서버 확장성 | 다중 서버 환경 동기화 | Redis Pub/Sub 도입 |
| High | 인증 갱신 | 토큰 만료 시 연결 유지 | 토큰 갱신 후 재연결 |
| Medium | 상태 동기화 | 클라이언트 상태 불일치 | 연결 시 상태 동기화 |

### 멀티에이전트 협력 제안

```
[intelligent-task-router]
  → [prompt-enhancer] (현재)
  → [parallel-task-executor] (백엔드/프론트엔드 병렬)
  → [iterative-quality-enhancer] (품질 검증)
```

### 자동 개선 루프 결과

| 반복 | 완전성 | 명확성 | 일관성 | 실행가능성 | 총점 | 상태 |
|------|--------|--------|--------|------------|------|------|
| 1차 | 4/5 | 4/5 | 4/5 | 3/5 | 15/20 | 개선 필요 |
| 2차 | 5/5 | 4/5 | 5/5 | 4/5 | 18/20 | 통과 |

**주요 개선 사항**:
- 확장성 고려사항 추가
- Phase 분리로 구현 순서 명확화
- 위험 완화 방안 구체화
```

**예상 소요 시간**: 60-120초

## 평가 기준

### 점수 체계

각 항목을 0-5점으로 평가합니다.

| 항목 | 가중치 | 평가 기준 |
|------|--------|-----------|
| GOLDEN 요소 | 30% | 복잡도에 맞는 요소가 포함되었는가 |
| Layer 구조 | 25% | 컨텍스트/지시/검증이 적절히 분리되었는가 |
| 프로젝트 컨텍스트 | 25% | 실제 프로젝트 정보를 정확히 반영했는가 |
| 위험 평가 | 20% | 잠재적 문제를 적절히 식별했는가 |

### 점수 해석

| 총점 | 등급 | 해석 |
|------|------|------|
| 4.5-5.0 | A | 우수 - 즉시 사용 가능 |
| 4.0-4.4 | B | 양호 - 약간의 보완 필요 |
| 3.0-3.9 | C | 보통 - 개선 필요 |
| 2.0-2.9 | D | 미흡 - 상당한 개선 필요 |
| 0-1.9 | F | 실패 - 재작업 필요 |
